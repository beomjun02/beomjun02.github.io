Date: 2026-05-29

What

Cron failed with claude-opus-4-8 unavailable. Fixed by pinning --model claude-opus-4-6 --effort xhigh. Then investigated pipeline performance — tested pre-running prefetch/dedup in bash vs letting LLM handle everything.

Root cause: model default changed

Fix: --model claude-opus-4-6 --effort xhigh pinned in run_all_users.sh.

Experiment: bash pre-run vs LLM-driven

Hypothesis: moving prefetch + dedup to bash (before LLM) should save ~1min.

ConditionSKILL.md §2TimeItems
OLD (LLM does prefetch)"run prefetch.py"8min13
NEW (bash pre-ran, LLM skips)"already done, just read"14min12

Result: opposite of expected. Pre-running was 75% slower.

Why?

Changing SKILL.md §2 from "run this script" to "data is already ready, just read it" caused the LLM to spend MORE time — likely re-interpreting the changed instructions, additional reasoning about what to skip vs read, and processing the overall larger/more complex SKILL.md.

The LLM's tool-call overhead for running prefetch.py (~1 bash call) is trivial compared to the cost of a more complex prompt.

Takeaway

Don't optimize LLM pipelines by moving work to bash. The LLM's context window cost from changed/added instructions can outweigh the time saved. Simpler, familiar instructions > fewer tool calls.

Node.js stdout buffering

Claude CLI (Node.js) buffers stdout when redirected to a file. Output is 0 bytes until process exit, then flushed all at once. This caused us to repeatedly kill processes thinking they were stuck.

Lesson: never kill based on 0 output — check ps for CPU usage instead. If CPU > 0%, it's working.

  • SKILL.md §2: reverted to LLM-driven prefetch (8min version)
  • Slack: plain URL instead of link text
004: Upvote System + Trending Page 2026-05-28 ~ 05-29 Feature addition

Date: 2026-05-28 ~ 05-29

Type: Feature addition

What

Per-item upvote button on individual user pages + trending section on the all-users page. Users vote by clicking ▲ on items they find valuable. Trending shows items with ≥20% of members upvoting, top 3 cap.

Architecture


[User clicks ▲] → POST to Cloudflare Worker → forwards to Apps Script → Votes sheet (upsert)
[Page loads]     → GET from Cloudflare Worker → Apps Script reads Votes → returns my_votes → button state restored
[All-users page] → GET trending from Worker → Apps Script aggregates → top 3 displayed

Why Cloudflare Worker?

Apps Script responds with a 302 redirect (script.google.com → script.googleusercontent.com). The intermediate redirect lacks CORS headers, blocking browser fetch from reading the JSON response. POST works with mode: 'no-cors' (fire-and-forget), but GET needs to read the response body.

Tried and rejected:

  • JSONP: Apps Script returns text/html on the redirect hop, causing x-content-type-options: nosniff to block script execution
  • localStorage-only: Works but no cross-device sync (phone ↔ laptop)
  • Direct fetch: CORS blocked on redirect

Worker proxy adds Access-Control-Allow-Origin: * and eliminates the redirect issue. Cloudflare free tier: 100K requests/day (we use ~50/day).

Vote storage: upsert

Each (user, date, itemId) has at most one row in the Votes sheet. Upvote inserts or updates. Unvote (toggle off) deletes the row. No append-only log — clean state.

Google Sheets auto-converts date strings to Date objects. _normalizeDate() converts back to YYYY-MM-DD for matching.

Item ID: URL hash

item_id = MD5(link URL)[:10]. Same news across different users produces the same ID even if titles differ (LLM generates slightly different titles per user). Ensures cross-user vote aggregation for trending.

Trending threshold

voters.length ≥ ceil(total_registered_users × 0.2) → top 3 cap. Currently 6 users → threshold = 2. Items with <2 upvoters don't appear on trending. Sorted by voter count, not net score.

No date filter — all votes aggregated (items naturally expire via 7-day history pruning).

Design decisions

Upvote-only (no downvote)

Downvote removed after initial implementation. Rationale:

  • Single button = easier mobile tap target
  • Positive signal sufficient for implicit interest tuning
  • "Not upvoted" ≈ implicit disinterest — no need for explicit negative
  • Simpler upsert logic (toggle on/off, not 3-state)

Server as source of truth

Button state restored from server on page load (via Worker GET). No localStorage dependency. Works across devices — vote on phone, see state on laptop. ~3s latency from Apps Script cold start — acceptable for now.

Vote → implicit interest pipeline

Votes feed into §0.5 (daily feedback processing). The LLM analyzes upvote patterns and updates interests.implicit[]:

  • ≥2 upvotes on similar topics → synthesize implicit preference
  • e.g., "3 upvotes on KV cache papers" → implicit: "Tends to prefer KV cache and inference efficiency papers"

Raw votes stay in Sheets but never enter LLM context directly — only the distilled implicit interests do. Keeps context bounded.

Safari iOS

  • :hover sticks after tap → wrapped in @media (hover: hover) so touch devices skip hover effect
  • pointer-events: none on triangle <span> inside button → ensures click always hits the button element

Files changed

  • scripts/post_html.py: vote button HTML/CSS/JS, --enable-votes flag, --subdir override, collect_users from config, announcement banner, section_order
  • scripts/feedback_webapp.gs: upvote/unvote POST → _upsertVote, _getVotes (JSONP, per-user filter), _getTrending (voter count threshold + cap), _normalizeDate
  • scripts/cloudflare-worker.js: CORS proxy for Apps Script
  • scripts/dedup.py: code-based dedup (prepare + update), summary archiving to cache/<date>.md
  • scripts/publish_devlog.py: devlog markdown → HTML, collapsible entries
  • SKILL.md: §0.5 vote processing, §2.5 dedup, §9 cache update, implicit hierarchy, Reference §4/§8

003: Effect of Implicit Interests on Output Quality 2026-05-28 Experiment

Depends on: Experiment 002 (effort level comparison — established Opus xhigh as optimal)

Motivation

The interests.implicit field in config.json holds fine-grained preference signals from user feedback (via the tune_interest action). Unlike explicit keywords that control *what* is selected, implicit interests are natural-language soft constraints.

This experiment measures: (1) how implicit interests affect output when applied only at the final summary stage, and (2) whether injecting them into the paper selection stage (Pass 2) as tiebreakers produces controlled, predictable selection changes — like LoRA fine-tuning over a base model.

Setup

  • Model: Opus 4.6, effort=xhigh (optimal from Experiment 002)
  • User profile: john (robotics-focused)

- Primary: VLA, robot foundation model, robot learning, world models, JEPA models, world action models

- Secondary: video generative models, diffusion modeling, multimodal generation models

  • Input: Same prefetched data as Experiment 002 (43 X tweets, 187 arXiv papers)
  • Control variable: interests.implicit field + placement in prompt

Analysis

Implicit operates on two layers

LayerEffectStrengthEvidence
Description (how)Adds actionable metadata (code availability, quantitative details)Strong0→11 items with code statements (A→B)
Selection (what, tiebreaker)Swaps marginal papers matching implicit preferencesControlled2 drops + 1 addition, core 9 unchanged (A→C)

Explicit > implicit hierarchy works

The tiebreaker framing successfully bounded implicit's influence:

  • Core papers unchanged: all 9 papers with strong keyword matches survived
  • Only marginal papers affected: Colosseum V2 (benchmark), Uni-LaViRA (navigation) — both had weaker keyword relevance
  • Replacement is sensible: HumanoidMimicGen (manipulation, robot learning) is objectively a better fit for john's profile

This confirms the LoRA analogy: implicit adjusts within the space defined by explicit keywords, never overriding strong matches.

Where implicit should NOT be applied

Pass 1 (title filter): Titles are 10-15 tokens. Implicit preferences like "prefer real-robot experiments" cannot be evaluated from titles. Applying implicit here would either have no effect or cause false positives — equivalent to full fine-tuning instead of LoRA.

Watch list: Status checks, not discovery. Implicit preferences are irrelevant.

Architecture recommendation (implemented in SKILL.md)

Prompt phrasing at Pass 2:

"Select top N most relevant papers based on explicit keyword match. When choosing among similarly relevant papers, use implicit preferences as tiebreakers — explicit keyword relevance always takes priority."

Prompt phrasing at Final:

"Generate the summary reflecting both explicit interests (keywords) and implicit preferences (user feedback). Explicit interests determine what to include; implicit preferences shape how items are described and fine-tune ranking among similar items."

Conclusion

Implicit interests are a dual-layer tool:

1. Description layer (strong, always active): adds actionable metadata like code availability

2. Selection layer (controlled, via tiebreaker): swaps marginal papers in a predictable, bounded way

The explicit > implicit hierarchy works as designed. With the tiebreaker framing, implicit acts as LoRA — lightweight adjustment on a stable base, never overriding strong keyword matches.

Current implicit value

With john's 2 existing implicit entries, the primary value is description enrichment (code availability). As implicit accumulates to 5+ entries via feedback, selection-level effects will become more meaningful. The architecture is ready for that growth.

001: Two-Pass Paper Filtering 2026-05-28 Experiment

Motivation

arXiv paper HTML pages are ~150KB each. With 4 categories, the agent reads ~585KB (~150K tokens, 75% of context window) just for papers. Most of this is HTML noise (nav, CSS, links) and unrelated papers. The agent processes everything in a single pass, which both wastes context and risks missing papers buried deep in the HTML.

Hypothesis

A two-pass approach — (1) title-only semantic filter, (2) abstract read for candidates only — should reduce token usage while improving recall by giving the LLM better signal-to-noise ratio.

Setup

  • User: daewon (primary: Language/LLM, secondary: Robotics)
  • Data: 2026-05-28 arXiv prefetch (187 papers across cs.RO/LG/CL/AI)
  • Cap: 3 papers per category, 2 max for secondary (Robotics)
  • Model: Claude Opus 4.6 for all conditions

Conditions

ABaselineAgent reads raw HTML (~150KB/category). Single-pass selection.
BTwo-pass (generous)Pass 1: all 187 titles → LLM selects candidates (no per-category limit). Pass 2: candidates' abstracts → final selection.
CTwo-pass (top-10)Pass 1: all 187 titles → LLM selects top 10 per category (40 total). Pass 2: candidates' abstracts → final selection.

Analysis

1. All baseline papers retained. Both B and C found the same 3 papers that A found. No regression.

2. Two-pass finds more relevant papers. A found 3, B found 9, C found 11. The baseline likely missed papers because 150KB of HTML in a single context pass causes attention to degrade on papers in the middle/end of the list.

3. Tighter Pass 1 (C) outperforms generous (B). Counter-intuitively, C found 11 papers vs B's 9, despite having a stricter Pass 1 (40 vs 66 candidates). The tighter filter gave Pass 2 better signal-to-noise, letting it make better selections. C also found 2 robotics papers that B missed entirely.

4. Token savings. A: 150K → C: 73K (2x total reduction). But the main agent context drops from 150K to ~0 — the savings there are far more impactful since main context also holds X tweets, ranking, and summary generation.

5. B vs C token trade-off. C uses 10K fewer tokens than B (73K vs 83K) while producing better results. The per-category cap in Pass 1 is strictly better.

Conclusion

Adopt condition C (two-pass with top-10 per category).

  • 2x total token reduction, ~100% main context reduction for papers
  • 3.7x more papers found (11 vs 3) with better category coverage
  • Tighter Pass 1 paradoxically improves quality — less noise for Pass 2

Implementation notes

  • Pass 1 and Pass 2 each run as subagents → main agent context stays clean
  • HTML → JSON parsing happens in prefetch.py (title + abstract extraction)
  • Pass 1 input = title-only JSON (~26KB, ~7K tokens regardless of paper count)
  • Pass 2 input scales with Pass 1 output (~1.5KB/paper with abstract)
